# API Design & Versioning

This page documents how the Prisma AIRS TypeScript SDK relates to the upstream OpenAPI specs, how public types are versioned, and which repository checks keep Zod schemas and examples aligned with implementation.

:::tip[Looking for method signatures?]
The TypeDoc-generated [API Reference](../reference/api/) is the authoritative listing of exported classes, functions, schemas, types, constants, and examples. This page covers the design policy behind that surface.
:::

## Semantic versioning before 1.0

The SDK is published as `@cdot65/prisma-airs-sdk`. The current package version in `package.json` is `0.12.0`, so the package is still pre-1.0.

Under pre-1.0 semver:

- Minor releases (`0.x.0`) may contain breaking changes.
- Patch releases (`0.x.y`) are used for bug fixes, internal changes, schema-drift corrections, docs/tooling updates, and additive changes that do not alter existing public types.

For this SDK, the main compatibility surface is the TypeScript API that consumers compile against. Removing or renaming an exported symbol, changing a method signature, or reshaping a typed response is treated as breaking.

## Source of truth for API shapes

OpenAPI specs checked into `specs/` are the source of truth for service shapes, and `src/models/*.ts` contains the runtime Zod schemas used to validate responses:

```text
specs/
  ScanService_Oct2025.yaml
  mgmt_service_docs.yaml
  AIRS-Model-Security-DataPlane.yaml
  AIRS-Model-Security-Management.yaml
  PaloAltoNetworks-AIRS-RedTeam-DataPlane.yaml
  PaloAltoNetworks-AIRS-RedTeam-management.yaml
  dlp/
    DataFilteringProfiles.yaml
    DataPatterns.yaml
    DataProfiles.yaml
    Dictionaries.yaml
```

The SDK also tracks patterns from the Python `pan-aisecurity` SDK where relevant, but this TypeScript package exposes more domains: scan, management/DLP, model security, and red teaming.

## Forward compatibility with `.passthrough()`

Most response schemas use Zod `.passthrough()`:

```ts
export const ApiKeySchema = z
  .object({
    api_key_id: z.string(),
    api_key_name: z.string().optional(),
  })
  .passthrough();
```

That means unknown response fields are preserved instead of stripped or rejected. Additive server-side fields can appear without breaking installed SDK versions, while known required fields and known field types still validate.

If the server changes the shape of a known field, the request layer throws `AISecSDKException` with `ErrorType.RESPONSE_VALIDATION`. The static and live checks below are designed to catch that class of drift.

## Static schema drift check

`npm run preflight` runs `scripts/preflight-schemas.ts`. It loads the OpenAPI specs, converts exported Zod schemas to JSON Schema, and reports drift.

```bash
npm run preflight        # strict; exits non-zero on unacknowledged drift
npm run preflight:warn   # reports drift but exits zero
```

The diff is intentionally asymmetric: Zod can omit optional OpenAPI fields because `.passthrough()` preserves additive fields at runtime, but required fields and shared field types are checked strictly. Intentional divergences are recorded in `scripts/preflight/allowlist.ts`.

The main CI workflow runs this strict preflight gate after format, lint, and typecheck.

## Live response audit

`npm run audit:live` runs `scripts/live-audit.ts` against read-only endpoints on a real tenant.

```bash
npm run audit:live
```

It requires the relevant `PANW_*` credentials and classifies probes as `ok`, `schema-drift`, `auth`, or `other-error`. A `schema-drift` result means a live API response failed the SDK's Zod schema and should be investigated before release.

## Generated API reference

The Docusaurus site consumes TypeDoc output generated from `src/index.ts`.

```bash
npm run docs:api
npm run docs:build
```

`typedoc.json` writes Markdown pages to `docs-site/docs/reference/api`. The Docusaurus sidebar has an autogenerated `apiSidebar` category for that directory, so new exported symbols appear after TypeDoc generation rather than manual navigation edits.

The docs deploy workflow runs:

1. `npm ci`
2. `npm run docs:check`
3. `npm run docs:api`
4. `npm --prefix docs-site ci`
5. `npm --prefix docs-site run build`

## Example coverage gate

`npm run docs:check` runs `scripts/check-example-coverage.ts`, which builds a TypeDoc reflection model and fails if any public method or exported function lacks an `@example` JSDoc tag.

```bash
npm run docs:check
npm run docs:check:warn
```

The corresponding test is `test/docs/example-coverage.spec.ts`.

## Release notes and changesets

User-facing changes are recorded under `.changeset/`. The publish workflow runs on a GitHub release, installs dependencies, checks format/lint/typecheck/test/build, and publishes to npm with trusted publishing (OIDC).
